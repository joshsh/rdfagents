Abstract

RDFAgents is a data model and protocol for real-time, peer-to-peer knowledge sharing on the Semantic Web. etc. etc. This specification defines etc. etc.

...contributes:

  • a statement-centric data model based on RDF and Named Graphs
  • a FIPA domain ontology for inter-agent information exchange
  • FIPA performatives and protocols for information sharing supporting privacy, provenance, trust, and information accountability in the context of social machines
  • integration of FIPA agent identity and Semantic Web identity
  • conventions for peer-to-peer information discovery

1 Introduction

... both Linked Data and RDF streams ...

RDFAgents adheres closely to Linked Data best practices while replacing the request-response, client-server model of HTTP with the asynchronous and peer-to-peer model of FIPA agents. In analogy to the four Linked Data principles, the basic principles of RDFAgents are as follows:

  1. Use Semantic Web URIs as names for things
  2. Use FIPA agent identifiers as names for agents
  3. When someone asks an agent about a URI, provide useful information using the standards (RDF, SPARQL)
  4. Include links to other URIs and quotations from other agents, so that they can discover related information.
For example, if an information consumer asks the agent named info@example.org about the thing named http://dbpedia.org/resource/Beijing, that agent should respond with an RDF description of the city of Beijing. The description should include links to related things such as China, Beijing universities and historical landmarks, and so on. Furthermore, if the description includes information from multiple sources, such as a news service news@foo.com or a weather service weather@bar.net, this should be indicated through the use of Named Graphs and graph quotation. This gives the user the option of contacting those sources directly in order to get more detailed or more up-to-date news or weather information about Beijing.

RDFAgents uses the least expressive subset, fipa-sl0, of FIPA's standard Semantic Language [FIPA SC00008I] for agent communication.

Note: message encoding is to be specified in the FIPA message envelope which encloses each ACL message.

AR.1 Message content

The propositional content of a message is expressed in a content language, for which RDFAgents uses both fipa-sl0 and a number of RDF document formats, depending on the context. For example, an RDFAgents agent uses fipa-sl0 to subscribe to a stream of updates about a Semantic Web resource, while the content of the updates it receives is expressed in TriG or N-Quads syntax.

RDFAgents' fipa-sl0 content is expressed in a domain ontology with the name rdfagents (see ___), which introduces vocabulary elements for queries and failure conditions. Throughout this specification, when a message is said to use the rdfagents domain ontology, it is understood to be expressed in the fipa-sl0 content language, and vice versa. This is indicated in each message through the use of the FIPA language and ontology parameters (see [FIPA SC00061G]).

AR.1.1 RDF content languages

RDFAgents defines the following RDF-based content languages:

  • rdf-ntriples: the language of expressions in N-Triples [N-Triples], a line-based, plain-text format for RDF graphs. In RDFAgents, every N-Triples triple represents a statement in the ground graph.
  • rdf-turtle: the language of expressions in the Turtle [Turtle] plain-text format. All statements belong to the ground graph.
  • rdf-xml: the language of expressions in RDF/XML [RDF/XML], the most widely supported RDF format. All statements belong to the ground graph.
  • rdf-nquads: the language of expressions in the N-Quads [N-Quads] format, which extends N-Triples with context. In RDFAgents, the context component of an N-Quads tuple MUST be a graph name. If a tuple has no context component, it represents a statement in the ground graph; otherwise, it represents a statement in the graph thus named.
  • rdf-trig: the language of expressions in the TriG [TriG] format, which expressly supports Named Graphs and RDF Datasets. The unnamed graph (if any) in a TriG document is identified with the ground graph in RDFAgents.
  • rdf-trix: the language of expressions in the TriX [TriX] XML-based format with support for graph names. Named and unnamed graphs are interpreted as in rdf-trig.

An RDFAgents platform MUST support at least one of the above languages, where supporting a language means that the platform can both parse and write the full range of valid expressions in the language to and from the corresponding RDF Datasets. However, an RDFAgents platform need not support all of the above languages. For the sake of interoperability, it is recommended that the rdf-xml and rdf-nquads languages be supported.

Agents MAY use additional RDF-based content languages which are understood, a priori, by both participants in a conversation. For example, an agent which receives a request for query results in the language rdf-json may choose to respond with a content expression in the Talis RDF/JSON format. It is the responsibility of the requester to validate the response against the expected format.

RDF content is implicitly grounded in the universal representational space of the Semantic Web.

AP.3 Quotation and syndication

Graph quotation serves a dual purpose in RDFAgents: it gives due credit to information providers, while also giving information consumers a means of contacting providers in order to verify syndicated information or to find additional information.

P.1 Query

The Query protocol allows an agent to request a single unit of information of another agent. It is based on the FIPA standard Query Interaction Protocol [FIPA SC00027H] identified by fipa-query.


Figure 1: Query protocol

The Query protocol involves two otherwise equal participants which assume the roles of a client requesting information from a server. A Query interaction begins with the client posing a query to the server as the content of a FIPA query-ref message, where the query is expressed in fipa-sl0 using an RDFAgents query form. Upon receipt of the query, the server either agrees or refuses to evaluate it. If the server refuses, it sends a refuse message to the client whose content is an explanation in the fipa-sl0 content language conforming to the RDFAgents domain ontology, completing the interaction. If the server agrees, it may choose to send an agree notification to the client. Otherwise, it proceeds directly to the evaluation of the query. Depending on the outcome of query evaluation, the server completes the interaction either with an inform-result message with an informational payload in one of the RDF content languages, or a failure message

P.1.1 The describe query form

This version of RDFAgents supports exactly one type of query, called describe, which accepts a single resource identified by a URI and returns an RDF dataset of information about that resource. This is similar to SPARQL's DESCRIBE query form in that the way in which the server selects an appropriate chunk of information "describing" the given resource is unspecified.

The client MAY request the query result to be expressed in a particular RDF format by using the :language parameter. If so, the server MUST either return results in the specified format, or raise an exception if the specified format is not supported. Otherwise, the query result MUST use any one of the RDF content languages explicitly listed above, where the payload of the message is expressed in the corresponding RDF format.

Example query: 

(query-ref
    :sender (agent-identifier
        :name http://fortytwo.net/agents/smith
	:addresses (sequence xmpp:smith@fortytwo.net))
    :receiver (set (agent-identifier
	:name http://example.org/agents/rdfnews
	:addresses (sequence xmpp:rdfnews@example.org)))
    :protocol fipa-query
    :conversation-id C976B710A5
    :language fipa-sl0
    :ontology rdfagents
    :content "(describe (resource \"http://dbpedia.org/resource/Beijing\") :language rdf-trig)")

Example query result:

P.1.2 Interpretation of query results

In RDFAgents, the inform-result message for the query result has the same propositional status as an inform message (see [FIPA SC00037J]): it is an assertion by the server that the ground statements in the RDF payload of the message are true. Furthermore, it is an assertion not only to the client which issued the query, but to all potential information consumers, making it similar to the act of publishing a document on the Semantic Web.

After receiving the query result from the server, the client MAY convey its informational content to any other agent, at any time, using the mechanism of graph quotation described above.

The Publish-Subscribe protocol enables event-driven streams from information providers to information consumers according to topics of interest. It is based on the FIPA standard Subscribe Interaction Protocol [FIPA SC00035H] identified by fipa-subscribe.

P.2.1 Interpretation of updates

As in the Query protocol, the inform-result messages containing updates have the same propositional status as an inform message: they are an assertion by the publisher, to all potential information consumers, that the ground statements in the RDF payload of the message are true. This provides a means of publishing data streams on the Semantic Web which is analogous to the publishing of RDF documents.

After receiving the update from the publisher, the subscriber MAY convey its informational content to any other agent, at any time, using the mechanism of graph quotation described above.

FC FIPA compliance

The FIPA Abstract Architecture [FIPA SC00001L] defines architectural elements of agent platforms and their relationships, and provides guidelines for the instantiation and interoperability of agent systems.

In order for a concrete architectural specification to be FIPA compliant, it must have certain properties. First, the concrete architecture must include mechanisms for agent registration and agent discovery and inter-agent message transfer. These services must be explicitly described in terms of the corresponding elements of the FIPA Abstract Architecture.

Any concrete implementation of RDFAgents, or RDFAgents platform, MUST realize the mandatory elements of the FIPA Abstract Architecture. The following subsections will describe additional constraints imposed by RDFAgents with respect to FIPA compliance, and discuss optional elements of FIPA which are not required by RDFAgents. All elements of FIPA not discussed in the following are to be considered independent of RDFAgents, and may be realized in implementation-specific ways.

FC.1 Agent identity

FIPA defines an agent identifier (AID) as an extensible collection of parameter-value pairs comprising a name or globally unique identifier for the agent, zero or more transport addresses at which the agent can receive messages, and zero or more resolvers or name resolution service addresses.

FC.1.1 Agent name

At creation or registration time, a FIPA agent MUST be given an agent name which has a durable relationship with the agent it identifies and which is chosen in such a way as to ensure global uniqueness. An agent name SHOULD have no semantics:

It should not encode any actual properties of the agent itself, nor should it disclose related information such as agent transport-description or location. It should also not be used as a form of authentication of the agent.

To facilitate the RDF description of agents, the name of an RDFAgents agent MUST be a Semantic Web URI which SHOULD be drawn from a URI space controlled by its owner. As such, it SHOULD be an HTTP URI associated with an RDF description of the agent which can be obtained by dereferencing the URI, in the manner of Linked Data.

The resource denoted by the agent name is implicitly an instance of foaf:Agent within RDFAgents and SHOULD be explicitly declared as such in RDF descriptions mentioning the agent.

The Semantic Web URI of a foaf:Person MAY be used as an agent name only if the RDFAgents agent and the person thus identified are identical for the purpose of information exchange. This may be the case, for example, when an agent acting as a personal assistant is given the task of sending and receiving messages on behalf of its owner. Note, however, that any discoverable contact information of the person which is not valid contact information of the agent may lead to unwanted and ineffectual messages from other agents. Additionally, it should be kept in mind that the contact information of an RDFAgents agent is shared freely, much as URIs are shared in Linked Data, while the sharing of personal contact information is subject to concerns of privacy and spam control.

FC.1.2 Transport addresses

A transport address is a physical address at which an agent can be contacted. It is usually specific to a particular protocol (for example, HTTP or XMPP). A given agent may support multiple methods of communication and therefore may have multiple transport addresses. The EBNF syntax of a FIPA transport address is the same as that of a URL given in [RFC 2396]. The transport addresses of an RDFAgents agent SHOULD be absolute URLs whose scheme part corresponds to a well-known messaging protocol (e.g. http or xmpp) and whose scheme-specific part serves to identify the recipient in an unambiguous, global fashion.

Every agent in an RDFAgents platform requires one or more transport addresses. If there are multiple agents in the platform, they MAY share the same transport address(es). The agent platform itself does not require a transport address and does not need to send or receive messages.

An agent's transport addresses SHOULD correspond with the contact information provided in authoritative RDF descriptions of the agent.

FC.1.3 Name resolution

RDFAgents does not require the use of FIPA's resolvers parameter of the agent ID (although it does not probihit its use). No specialized lookup services are provided for agent names. Instead, transport addresses are inferred from the RDF descriptions of agents. An RDFAgents platform MUST be capable of reading RDF descriptions and applying at least the following rule for the mapping of contact information to transport addresses.

Every RDFAgents platform MUST include a list of transport-level services with which it is capable of interoperating. This list MAY contain as few as a single service, identified by the URL of its home page. Each service MUST be bound to a transport protocol such that, when the platform is provided with the service-specific identifier of a recipient, it can send a message to that recipient using the associated protocol.

While there are many possible conventions for expressing agent contact information in RDF, only one is required to be supported by an RDFAgents platform, namely, the use of the vocabulary elements foaf:account to associate an agent with a account, and foaf:accountServiceHomepage and foaf:accountName to associate the account with a service and a service-specific identifier, respectively.

For example, the following RDF description provides sufficient information for the sending of a message over XMPP to the agent identified by rdfnews@example.org on Jabber: 

{
    <newsAgent> a foaf:Agent ;
        foaf:name "RDF News Agent" ;
        foaf:account [
            a foaf:OnlineAccount ;
            foaf:accountServiceHomepage <http://www.jabber.org/> ;
            foaf:accountName "rdfnews@example.org" ] .
}

In order to make use of this information, an RDFAgents platform must interoperate with the service identified by http://www.jabber.org/, which involves knowing that it is an XMPP-based service, supporting the XMPP protocol, and having a Jabber account.

FC.2 Agent management

FIPA Agent Management [FIPA SC00023K] provides the normative framework within which FIPA agents exist and operate. It establishes the logical reference model for the creation, registration, location, communication, migration and retirement of agents. A mandatory component of this model is the agent management system, or AMS, which exerts supervisory control over access to and use of the platform. It provides agent identifiers to agents and maintains a directory of identifiers, providing a white pages service for all agents.

However, the AMS is merely an internal component of a conforming RDFAgents implementation; this specification is concerned only with peer-to-peer communication between platforms (as opposed to intra-platform communication), and there is no operation within RDFAgents which requires the sending of a message to the AMS of a remote platform.

FC.2.1 Service discovery

RDFAgents neither requires nor prohibits the use of FIPA's optional directory facilitator, which maintains an index of agents in a platform and provides a yellow pages service for agents. The role of the directory facilitator is considered to be fulfilled in RDFAgents by the "word of mouth" mechanism of service discovery via RDF descriptions.

FC.4 Inter-agent message exchange

All FIPA agents must have access to at least one message transport service, or concrete implementation of a protocol for sending and receiving messages. In RDFAgents, each agent MUST provide at least one valid transport address, for a supported transport service, in each message it sends. Additionally, the agent SHOULD provide corresponding contact information in each RDF description of itself.
